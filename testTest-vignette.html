Intro

The sole purpuse of this tiny packacge (only 4 lines of code) is to demonstrate a difference in verdicts made by 3 kinds of tests and to provide a remedy.

The package used an environment with data that may be manipulated. And, in order to get predictable results each test file should start with the same ‘factory’ data. However, achieving this by detaching, unloading and re-requiring the package at the start of each test file will causes a test problem.

Different ways of testing rersulted in different verdicts.

Overview

  1. The intended behaviour of this package is demonstrated.

  2. The ‘setMenu’ function is tested with a simple use of ‘testthat’ This works well.

  3. The ‘setMenu’ function is tested with ‘rcmdcheck()’ Which works well.

  4. The ‘setMenu’ function is tested with ‘devtools’ Which caused a problem.

  5. Concluding considerations

  6. A simple remedy

Demonstration of the testest package

The ‘factory set’ menu is printed, then it is set to the new menu and the new menu should be printed correctly

printMenu ()
#> [1] "spam"
setMenu ("Bacon")
printMenu ()
#> [1] "Bacon"

Conclusion: it works perfectly.

The test file use for all tests

The package test-file ‘testTestTest.R’ reads:

require(testthat)
#> Loading required package: testthat

# laoding a fresh package
if (isNamespaceLoaded ("testTest")) detach (package:testTest, unload = TRUE)
require (testTest)
#> Loading required package: testTest
testEnvir <- testTest:::testEnvir

test_that ("testTestTest", {
    expect_equal (testEnvir$menu, "spam")
    testEnvir$menu <- "eggs"
    expect_equal (testEnvir$menu, "eggs")

    setMenu ("bacon")
    expect_equal (testEnvir$menu, "bacon")   # fails with devtools::test() only
})
#> Test passed

Note that the testTest package is detached, unloaded, and re-required at the beginning. This is to make shure that the test is preformed on testEnvir data with ‘factory’ settings, i.e. on data that is not changed with ‘setMenu’ functions.

However as shown below, this will cause a severe problem.

It looks like devtools:::test_active_file() keeps using the ‘old’ testEnvir. i.e. the one that exists when the function in invoked. And, that ‘setMenu (“bacon”)’ used - as required for this package - the new environment made available after re-requiring the ‘testTest’ package.

Concluding considerations

As devtools:::test_active_file() may produce a negative verdict for expect_… cases may be ok for the package at hand and for rcmdcheck or sourcing a test file, it may impose unnecessary drawbacks for some package developers. This holds especially in case the package contains data that may extended by users who may e.g. add new objects and/or add/modify object attributes and/or set/modify user preferences (e.g. for specific objects).

However there does exists a simple work around.

A simple remedy

Make a copy of all vulnerable ‘factory’ data in the environment under test and restore it at the beginning of each test file.

Add, e.g. a function like this in “testthat/helper.R”

resetMyData <- function () {
    if (myBackup doesn't yet extsts) {
        backup vulnarable data  # will be excecuted for the first testfile only
    } else {
        restore from backup
    }
}

And add resetMyData() to the beginning of each test file.
